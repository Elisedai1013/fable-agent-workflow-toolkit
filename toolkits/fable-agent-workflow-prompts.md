# 《A Field Guide to Fable》｜六组 Agent 协作方法

[← 返回仓库首页](../README.md)

## 来源与使用说明

这 6 组方法根据 Thariq Shihipar 在 AI Engineer World’s Fair 2026 的演讲 [A Field Guide to Fable](https://www.youtube.com/watch?v=9fubhllmsBU) 整理；活动与讲者信息可在 [AI Engineer World’s Fair 2026 官方议程](https://www.ai.engineer/worldsfair/schedule?q=Field%20Guide%20to%20Fable) 核验。它们适合需要把非简单任务交给 Agent，同时希望保留关键判断与责任的产品、设计和工程 Builder。

原演讲讨论了如何给更强的模型松绑、寻找未知点，以及为什么人仍需留在关键决策循环中。`blind spot pass`、`brainstorms and prototypes`、`interviews`、`references`、`implementation notes` 和 `quiz me` 来自 Thariq 的分享；下面的六份中文可复制模板、输出结构、扩展规则与组合流程由「Elisedai在创造」根据这些观点重新整理，并非 Thariq 或 Anthropic 发布的官方工作流。

把这些做法迁移到其他 Agent，是 AI Builders 解读做出的编辑性泛化。具体表现会随模型、工具权限、代码库和运行环境变化。

使用时，把方括号中的内容替换成你的实际信息。

## 快速选择

| 你现在遇到的问题 | 使用方法 | 主要产出 |
| --- | --- | --- |
| 害怕需求里漏掉关键问题 | 盲点扫描 | 未知点清单与修订后的任务说明 |
| 说不清自己真正喜欢哪种方向 | 差异化原型 | 四个可比较方向及其取舍 |
| 关键判断还藏在脑中 | AI 逐题访谈 | 已确认、仍未知、关键决策与执行规格 |
| 已有参考，但不想机械复制 | 参考实现 | 语义重实现与差异、假设清单 |
| Agent 会自主执行较长任务 | 执行偏差记录 | 可追踪的 implementation notes |
| 准备审查、合并或发布 | 完成后反向测验 | 总结、验证状态、风险与理解缺口 |

## 工具一｜盲点扫描

### 来源关系

Thariq 用“已知与未知”矩阵说明：真正会改变方案的，往往是人还没意识到的问题。他在原分享中直接使用了 `blind spot pass`，并以不熟悉的认证模块为例寻找 unknown unknowns。本期在此基础上增加了五个检查领域、固定输出结构与任务改写步骤。

### 适用对象

正在把不熟悉、跨领域或风险较高的任务交给 Agent 的产品、设计或工程 Builder。

### 什么时候使用

适合需求看起来已经完整，但你对业务、代码库或某个领域并不熟悉时。它的目的不是润色需求，而是主动挑战需求。

### 输入

- 项目或产品背景；
- 当前具体任务；
- 你不熟悉的领域或模块；
- 已知的范围、权限和不可变约束。

### 可复制提示词

~~~text
我正在 [项目/产品] 中处理 [具体任务]，但目前对 [领域/模块] 不够熟悉。

请先不要执行，先做一次盲点扫描：
1. 找出我可能完全没有意识到、但会改变方案的问题；
2. 优先关注架构、数据、安全、成本和用户体验；
3. 按“盲点—为什么重要—需要补充什么信息”输出；
4. 最后根据扫描结果，帮我改写一版更完整的任务说明。
~~~

### 输出

一组会影响方案的未知点，以及一份更完整的任务说明。

### 使用边界

AI 找到的是待验证问题，不应把推测直接当成事实。涉及安全、法律、权限或高成本决策时，必须回到可核验资料和专业判断。

### 下一步

核验真正会改变方案的盲点，把已确认内容写回任务说明；仍需人取舍的问题进入逐题访谈，需要比较偏好的部分进入差异化原型。

## 工具二｜差异化原型

### 来源关系

Thariq 用 `brainstorms and prototypes` 暴露 unknown knowns，并举例让模型生成四个设计决策差异很大的 HTML 页面。本期进一步要求差异覆盖信息结构、交互和视觉，并增加取舍说明、适用场景与三个反馈问题。

### 适用对象

目标已经明确，但暂时说不清具体结构、交互或视觉偏好的 Builder。

### 什么时候使用

适合你知道目标，却还无法准确描述自己想要的结构、交互或视觉方向时。

### 输入

- 目标用户和核心目标；
- 要探索的页面或功能；
- 必须保留的内容与技术约束；
- 评估各方向时最关心的标准。

### 可复制提示词

~~~text
我想为 [目标用户] 设计一个用于 [目标] 的 [页面/功能]，但还不确定最合适的方向。

请先不要直接做最终稿：
1. 生成 4 个方向明显不同的 HTML 原型；
2. 差异必须体现在信息结构、交互方式和视觉表达，而不只是颜色；
3. 说明每个方案的核心取舍与适用场景；
4. 最后提出 3 个问题，帮助我根据反馈确定最终方向。
~~~

### 输出

四个真正不同的方向，每个方向的核心取舍、适用场景和待回答问题。

### 使用边界

原型用于暴露偏好，不等于经过用户验证的最终方案。比较时不要只看视觉新鲜感，还要检查结构、交互和信息理解的差异。

### 下一步

选择一个方向或组合出新方向，记录选择理由；再用逐题访谈补齐会影响实现的产品与技术决策。

## 工具三｜AI 逐题访谈

### 来源关系

Thariq 建议让 Claude `interview me`，并强调优先询问那些会改变架构的问题。原分享也举过一次要求模型提出 40 个问题的例子；“一次只问一个”是本期为减少回答负担、方便逐步对齐而增加的交互规则，并非 Thariq 的原话。

### 适用对象

需要把脑中的隐含判断变成执行规格，但不希望一次面对大量问题的 Builder。

### 什么时候使用

适合任务目标比较模糊，或者关键产品和技术决策仍藏在脑中、没有写进需求时。

### 输入

- 任务或产品目标；
- 已经确认的信息；
- 已知的范围、技术和交付约束；
- 你当前最纠结的决策。

### 可复制提示词

~~~text
我准备完成 [任务/产品目标]。请先不要给方案，而是对我进行需求访谈。

要求：
1. 每次只问一个问题，等我回答后再继续；
2. 优先询问那些答案不同，就会改变产品范围、技术架构或交付方式的问题；
3. 不要重复询问我已经明确提供的信息；
4. 访谈结束后，整理“已确认、仍未知、关键决策”并生成一份执行规格。
~~~

### 输出

一份区分“已确认、仍未知、关键决策”的访谈结论，以及一份可执行规格。

### 使用边界

问题数量不是目标，找到会改变整体方案的决策点才是目标。AI 负责追问和整理，不能替人回答价值取舍。

### 下一步

由人确认执行规格，把仍未知的事实交给调查或验证，把仍未决定的价值取舍保留为明确的人类决策点。

## 工具四｜参考实现

### 来源关系

Thariq 把 `references` 称为“给 Claude 另一张地图”：参考可以来自不同系统或语言，例如用 HTML mock-up 表达 React 组件目标。本期增加了语义重实现、差异与假设清单，以及版权和授权边界。

### 适用对象

已有代码、页面或产品可以表达目标效果，但需要跨技术栈重实现或避免机械复制的 Builder。

### 什么时候使用

适合已有页面、代码或产品能够表达目标效果，但你不希望 AI 逐像素或逐行复制时。

### 输入

- 可访问的参考代码、页面或产品；
- 希望保留的核心效果；
- 目标技术栈或目标产品；
- 授权、兼容性和品牌约束。

### 可复制提示词

~~~text
下面的 [代码/页面/产品] 代表我希望实现的核心效果：[粘贴或提供路径]。

请：
1. 先理解它的行为、结构、交互和设计意图；
2. 提取必须保留的语义，而不是逐行或逐像素复制；
3. 在 [目标技术栈/目标产品] 中重新实现；
4. 明确列出与参考实现的差异、假设和原因；
5. 遇到无法确定的部分，先向我提问。
~~~

### 输出

重实现结果，以及与参考的差异、假设和原因清单。

### 使用边界

提供参考实现不代表拥有复制其代码、设计或素材的权利，使用前仍需确认许可。优先保留行为与设计意图，而不是逐行或逐像素复制。

### 下一步

先审查提取出的语义、差异和授权条件，确认无误后再进入实现；无法确认的部分应回到原作者、规范或负责人处核验。

## 工具五｜执行偏差记录

### 来源关系

Thariq 建议在模型遇到 unknown 时留下 `implementation notes`，以便看见偏差发生在哪里以及原因。本期增加了固定记录字段、优先选择保守可逆方案，以及重大风险必须暂停确认的安全规则。

### 适用对象

允许 Agent 自主推进较长任务，同时需要让关键偏差可追踪、可复盘的 Builder。

### 什么时候使用

适合让 Agent 自主推进较长任务，同时希望关键偏差可追踪、可复盘时。

### 输入

- 原计划或执行规格；
- Agent 可以自主决定的范围；
- 必须停下确认的权限和风险边界；
- 偏差记录文件路径。

### 可复制提示词

~~~text
执行任务时，请持续维护一份 implementation-notes.md。

如果遇到迫使你偏离原计划的边界情况：
1. 优先选择保守、可逆的方案；
2. 在“Deviations”下记录原计划、实际偏差、原因、影响和后续动作；
3. 如果不涉及安全、权限或不可逆变更，可以记录后继续；
4. 如果会显著改变范围或架构，请暂停并向我确认。
~~~

### 输出

一份持续更新的偏差记录，每项都包含原计划、实际偏差、原因、影响和后续动作。

### 使用边界

记录不能扩大 Agent 的权限。安全、权限、外部发布、不可逆变更或显著的范围与架构变化，仍需人明确确认。

### 下一步

定期审阅偏差记录：可以接受的偏差写回新基线；会改变范围、架构或风险等级的偏差必须由人决定后才能继续。

## 工具六｜完成后反向测验

### 来源关系

Thariq 会让 Fable `quiz me`，以确认自己理解发生了什么，并能在创建 PR 或合并时解释这份工作。本期增加了结构化总结、固定五道题、暂不显示答案和识别理解缺口等执行细节。

### 适用对象

准备审查、合并或发布 Agent 产出，需要确认自己仍理解关键修改与风险的 Builder。

### 什么时候使用

适合 Agent 已完成实现，而你需要确认自己仍理解关键修改、风险和发布条件时。

### 输入

- 已完成的代码、配置或产品变更；
- 原需求和验收标准；
- 已运行的测试和验证结果；
- 已知的偏差、未验证项和风险。

### 可复制提示词

~~~text
任务完成后，请生成一份可阅读的总结报告，包含：
1. 修改了什么，以及为什么；
2. 关键决策、执行偏差与仍然存在的风险；
3. 如何验证结果，以及哪些部分尚未验证；
4. 我在合并或发布前必须理解的内容。

最后根据本次工作出 5 道测验题。在我回答之前不要给出答案；根据我的回答指出理解缺口。
~~~

### 输出

变更总结、关键决策、验证状态、未解风险和五道测验题。

### 使用边界

答对测验不能替代代码审查、自动化测试、安全检查或真实用户验证。如果 Agent 无法提供可核验证据，不能因为报告看起来完整就认定任务通过。

### 下一步

先回答测验并回到变更、测试和偏差记录核验理解缺口；只有审查与验证条件真正满足后，才进入合并或发布。

## 推荐组合

下面的六步组合是「AI Builders 解读 01」根据原演讲做出的编辑性串联，不是 Thariq 命名的正式六步法：

1. 先做盲点扫描；
2. 需要探索方向时生成差异化原型；
3. 用逐题访谈补全关键决策；
4. 用参考实现传递具体语义；
5. 实施过程中持续记录偏差；
6. 完成后通过总结和测验确认人仍在决策循环中。

核心原则不是让人控制 Agent 的每一步，而是让 Agent 执行、让人理解和校准关键决策，并对结果负责。
